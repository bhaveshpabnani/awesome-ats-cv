# Bullet Writing

## Bullet formula

Use this structure when evidence exists:

`Action verb + work done + method/technology + scale/constraint + measurable result`

Useful variants:

- `Action + technical object + technology + measurable impact`
- `Action + business/user problem + implementation + result`
- `Action + model/system + baseline to final metric`
- `Action + ownership scope + production quality signal`

## STAR and XYZ models

STAR:

- Situation: what problem or context existed
- Task: what needed to happen
- Action: what the candidate personally did
- Result: what changed because of the action

XYZ:

- Accomplished X
- Measured by Y
- By doing Z

Prefer compact bullets that imply STAR/XYZ without becoming long narratives.

## Content rules

- Start every bullet with a strong action word.
- Avoid repeating action verbs in nearby bullets.
- Keep each bullet to one idea.
- Replace adjectives with evidence: latency, accuracy, volume, records, users, cost, revenue, time saved, rank, scale, tests, services, repos, endpoints, files, release version, deployment count, or data size.
- Never invent metrics. If a metric is not known, use defensible scale from the codebase or source material, such as number of services, integrations, records, endpoints, or release artifacts.
- Include technical terms and product keywords that match the target role: frameworks, cloud services, databases, algorithms, runtime surfaces, models, protocols, and deployment tooling.
- Bold the most important keywords inside bullets, such as `Aurora`, `Redis`, `AWS EC2`, `DynamoDB`, `LangGraph`, `FastAPI`, `Kafka`, `Playwright`, `MSE`, model names, and named systems.
- Prefer recruiter-readable impact over internal variable names. Replace implementation-only terms such as raw column names or local aliases with scale, architecture, latency, cost, records processed, multi-tenancy, API surface, or customer-facing result.
- Every bullet should fill most of the page width. If a bullet renders short, enrich it with real scope or result instead of adding vague filler.
- Highlight candidate contribution, not the prestige of the event or group.
- Avoid first-person pronouns.
- Avoid vague terms such as various, multiple, several, many, helped, worked on, responsible for.
- Do not include points that the candidate cannot explain in interviews.
- Avoid lower-level duplicate achievements when a higher-level achievement proves the same skill.

## Technical bullet examples

- Reduced search latency from 4s to 843ms by batching DynamoDB queries through a 20-thread Guice ExecutorService
- Improved forecast Mean Squared Error from 0.0096 to 0.0046 using H2O.ai stacked ensemble AutoML
- Shipped macOS/Windows desktop release artifacts with checksums, CI gates, Homebrew, Scoop, Chocolatey, and WinGet manifests
- Built a Retrieval-Augmented Generation (RAG) pipeline over 2.7M+ documents using BM25, vector search, FAISS, and reranking

## Editing checklist

- Does the first word carry force?
- Does the bullet show individual contribution?
- Does it include a metric or verifiable scale?
- Does it contain exactly one idea?
- Does it include target-relevant technical keywords?
- Are only the strongest keywords bolded, without bolding the whole sentence?
- Does it render on one line and occupy roughly 90-99 percent of the available width?
- Can the candidate explain it in detail?
- Is the grammar clean after removing the full stop?
- Is the wording specific enough for a recruiter outside the institute?
